## 一、LLM 驱动的工具调用系统

### 1.1 架构设计

采用 OpenAI function-calling 协议，LLM 自动决定是否调用工具，前端只做渲染展示。

```
用户消息 → POST /api/chat
  ↓
后端构建 messages（system + 历史 + 当前 query），传入 tools 定义
  ↓
LLM 自主判断：
  ├─ 不需要工具 → 流式文本回答
  └─ 需要工具 → finish_reason: tool_calls
       ↓
    ① SSE event: tool_call { name, arguments }
    ② 后端执行工具
    ③ SSE event: tool_result { name, result }
    ④ 结果回传 LLM → 第二轮对话 → 最终文本回答
```

### 1.2 核心实现

**后端工具执行循环**（`apps/server/src/chat/chat.service.ts`）：

- 多轮 tool calling 循环（最多 5 轮），前 3 轮传 tools，最后 2 轮不传强制输出文本
- 工具结果通过标准 OpenAI 格式回传：`assistant { tool_calls: [...] }` + `tool { tool_call_id, content }`
- 工具执行间隙通过 `tool_progress` SSE 事件推送进度状态

**前端工具渲染**（`apps/web/src/components/ToolRenderer/`）：

- `ToolRegistry` 全局单例管理工具定义（`register` / `get` / `list` / `has`）
- `TOOL_CONFIGS` 配置表控制每种工具在 UI 中的展示方式（one-line / collapsible / search-results）
- `ToolStore`（MobX）管理工具调用状态（running → completed → error）
- `ToolRenderer` 根据工具名自动路由到正确的显示组件

### 1.3 扩展新工具（三步注册）

```ts
// ① 在 builtin.ts 注册
toolRegistry.register({ name: 'xxx', handler: async (input, ctx) => { ... } })

// ② 在 configs.ts 添加显示配置
TOOL_CONFIGS.xxx = { input: { type: 'one-line', ... }, result: { type: 'collapsible', ... } }

// ③ 在 Chat 页面的 SSE 回调中推入 toolStore
toolStore.addCall('xxx', args)
```

## 二、联网搜索（Web Search）

### 2.1 搜索引擎

对接博查 AI 搜索 API（`https://api.bocha.cn/v1/web-search`），通过 `SEARCH_API_KEY` 环境变量配置。

**WebSearchService**（`apps/server/src/chat/web-search.service.ts`）：

- 请求体：`{ query, count: 5, freshness: 'noLimit', summary: true }`
- 响应解析：`data.webPages.value[] → { url, name, snippet, siteName, siteIcon, dateLastCrawled }`
- 业务错误码检查：`data.code !== '200'` 时自动降级
- 环境变量未配置时自动跳过，不阻塞对话

### 2.2 前端展示

- 搜索进行中：ToolRenderer 显示旋转动画 + "正在执行 web_search…"
- 搜索完成：按钮 "找到 N 条结果"，点击后悬浮卡片右侧展示去重后的结果列表
- 搜索结果按 URL 去重，聚合多次搜索调用的结果

## 三、上下文管理与自动压缩

### 3.1 上下文历史传递

前端在发送消息前提取当前会话的所有历史消息，通过 `history` 字段传给后端：

```ts
const history = conversationStore.messages.map(m => ({
  role: m.role as 'user' | 'assistant',
  content: m.content,
}))
```

后端构建 LLM messages：`system → 历史 → 当前 query`

### 3.2 滑动窗口截断

基于 token 估算的滑动窗口截断策略：

| 参数 | 值 | 说明 |
|------|-----|------|
| 模型上下文上限 | 128K | deepseek-v4-pro |
| 历史 token 预算 | ~102K | 80% 上下文窗口 |
| 截断策略 | 从最早消息开始丢弃 | 保留最近的消息 |
| Token 估算 | ASCII 0.25 / CJK 1 | 偏保守估算 |

### 3.3 自动摘要压缩（Compact）

当历史消息占用的 token 数达到预算的 70% 时，自动触发摘要压缩：

- 保留最近 6 条原始消息不压缩
- 早期消息通过 LLM 非流式调用生成 ≤500 字中文摘要
- 摘要以 system 消息形式插入上下文：`[更早的对话摘要]\n{summary}`
- 摘要后仍超预算时，滑动窗口兜底截断

## 四、多模态文件上传管线

### 4.1 系统架构

```
选文件 → blob URL 本地即时预览
  ↓
MediaStore.addItem() → status: 'pending'
  ↓
uploadFile():
  ├─ ① 计算文件哈希（SHA-256）
  ├─ ② POST /api/upload/check → 秒传命中 → completed（0 字节上传）
  └─ ③ XHR 上传（FormData + 进度回调）
       POST /api/upload → multer 接收 → MD5 登记 → 存 public/uploads/ → 返回 URL
  ↓
status: 'completed' + url
  ↓
发送消息: multiMedias 含文件 URL
  ↓
后端 buildMultimodalContent() 构建 OpenAI 多模态 content 数组
```

### 4.2 状态机设计

`MediaStore`（MobX，5 状态）：

```
pending → compressing → uploading → completed
                                  → error
```

每个文件独立追踪状态、上传进度（0-100%）、错误信息。

### 4.3 关键特性

| 特性 | 实现 |
|------|------|
| 哈希秒传 | POST `/api/upload/check { md5 }` → 命中则 0 字节上传 |
| 上传进度 | `XMLHttpRequest.upload.onprogress` 实时回调 |
| 本地预览 | `URL.createObjectURL()` 即时显示缩略图 |
| 多文件并发 | 每个文件独立上传管线，非阻塞 |
| JWT 鉴权 | 上传接口需要 Bearer Token |
| 文件大小限制 | 20MB（multer limits） |
| 支持类型 | image/\*、PDF、DOC、DOCX、TXT |

### 4.4 后端实现

**UploadController**（`apps/server/src/upload/upload.controller.ts`）：

- `POST /api/upload` — multer FileInterceptor，diskStorage 存临时文件
- `POST /api/upload/check` — MD5 哈希查询，秒传判断

**UploadService**（`apps/server/src/upload/upload.service.ts`）：

- 文件保存后用 MD5 前 8 位 + 扩展名重命名，避免路径冲突
- 内存哈希映射表（生产环境可改为数据库表持久化）
- 静态文件托管：`useStaticAssets(public, { prefix: '/' })`

### 4.5 前端组件

- **MediaPreviewItem**：图片缩略图 + 上传遮罩（旋转 + 百分比）；文件图标 + 文件名 + 状态指示器
- **MediaPreviewList**：输入框上方横向排列的附件预览列表，每个文件可独立删除

## 五、流式聊天优化

### 5.1 生产者-消费者模型

`streamChatMessage`（`apps/web/src/service/chat/index.ts`）：

- **生产者**：`readLoop` 全速消费 SSE 事件（`EventSourceParserStream`），token 累积到 `cachedContent`
- **消费者**：`requestAnimationFrame` 节流，每帧最多 flush 一次 token 更新
- **控制事件**（finish / error / usage / tool_call）立即入队，不经过 RAF，保证时延

### 5.2 事件类型

| SSE 事件 | 前端处理 |
|----------|---------|
| `message` → token | RAF 聚合 → MobX `appendToken` |
| `tool_call` | `toolStore.addCall` → ToolRenderer 渲染 |
| `tool_progress` | `toolStore.setProgress` → 显示"正在执行…" |
| `tool_result` | `toolStore.completeCall` → 状态变更为 completed |
| `finish` | 流结束，清理 RAF |
| `error` | 错误提示 + 回滚消息 |

## 六、多模态消息构建

后端 `buildMultimodalContent()` 根据附件类型构建 OpenAI 格式的 content：

- 无图片/文件时：返回纯文本字符串
- 有图片时：构建 `[{ type: 'text', text }, { type: 'image_url', image_url: { url: '...' } }]` 数组
- 历史消息仅传文本（不含 base64 图片，避免上下文膨胀）

## 八、技术架构

### 8.1 目录结构

```
apps/
├── server/                        ← NestJS 后端 (port 8081)
│   ├── src/chat/                  ← LLM 对话（流式 + tool calling）
│   ├── src/upload/                ← 文件上传（multer + MD5 秒传）
│   ├── src/auth/                  ← JWT 鉴权
│   ├── src/chat-history/          ← 对话历史持久化
│   └── src/lib/logger.ts          ← 本地日志工具
│
└── web/                           ← React 前端 (port 8000)
    ├── src/pages/Chat/            ← Chat 页面
    │   ├── components/            ← 页面专属组件
    │   └── index.tsx              ← 页面入口
    ├── src/components/            ← 通用组件（Layout, ToolRenderer）
    ├── src/controller/stores/     ← MobX 状态管理（conversation, tool, media）
    ├── src/service/               ← API 层（chat, api, tools）
    └── src/route/                 ← HashRouter 路由
```

### 8.2 核心设计模式

| 模式 | 应用场景 |
|------|---------|
| 状态机 | 文件上传（5 状态）、工具调用（3 状态） |
| 生产者-消费者 | SSE 流式消费（RAF 节流 + Promise notify） |
| 注册表 | ToolRegistry / TOOL_CONFIGS |
| AsyncGenerator | 服务端 LLM token 流式产出 |
| 策略模式 | ToolRenderer 根据工具名路由不同渲染 |
| 滑动窗口 | 上下文 token 预算截断 |

### 8.3 设计原则

- **LLM 驱动**：工具调用由 LLM 自主决策，前端只注册和渲染
- **CSS transition 优先**：动画用 Tailwind 声明，避免 JS 状态管理
- **渐进降级**：搜索 API / 上传服务不可用时自动降级，不阻塞对话
- **关注点分离**：数据获取（service）、状态管理（store）、UI 渲染（component）各司其职
