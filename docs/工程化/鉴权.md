# 企业级鉴权方案

---

## 一、三层架构总览

```
┌──────────────────────────────────────────────────┐
│  Layer 1: API 鉴权                                │
│  JWT accessToken (15min, Bearer header)           │
│  → 接口身份校验，短期有效，减少泄露窗口               │
├──────────────────────────────────────────────────┤
│  Layer 2: 自动刷新                                │
│  refreshToken (7d, HttpOnly Cookie + Rotation)    │
│  → 无感续期，每次刷新旧 token 立即作废（防重放）      │
├──────────────────────────────────────────────────┤
│  Layer 3: 会话/设备管理                           │
│  数据库记录设备指纹 + 登录历史 + 远程踢下线          │
│  → 异常登录检测、用户主动管理会话                    │
└──────────────────────────────────────────────────┘
```

---

## 二、Layer 1 — API 鉴权（accessToken）

### 签发

- 登录成功后，签发短期 **accessToken**（15 分钟），通过 HTTP `Authorization: Bearer xxx` 传递
- JWT payload 包含 `{ sub, username, jti }`，`jti` 用于后续黑名单吊销
- Secret 必须从环境变量注入，**严禁硬编码 fallback**

```typescript
// ❌ 禁止 — 硬编码 fallback 隐藏配置缺失
const secret = process.env.JWT_SECRET ?? 'yes-jwt-secret'

// ✅ 正确 — 启动即报错，不允许静默降级
const JWT_SECRET = process.env.JWT_SECRET!
if (!JWT_SECRET) throw new Error('JWT_SECRET 环境变量未设置')
```

### 校验

- 后端通过 `PassportStrategy` 从 `Authorization` header 提取并验证
- 验证通过后将 `req.user = { userId, username }` 注入上下文
- Guard（`JwtAuthGuard`）挂载到需要保护的路由，未携带有效 token 返回 401

### 登录频率限制

防止暴力破解：

```bash
npm i @nestjs/throttler
```

```typescript
import { Throttle } from '@nestjs/throttler'

@Throttle({ default: { limit: 5, ttl: 60000 } }) // 每分钟最多 5 次
@Post('/login')
async login(...) {}
```

### 密码强度策略


## 三、Layer 2 — 自动刷新（refreshToken）

### 存储：HttpOnly Cookie（非响应 body）

refreshToken 绝不暴露给 JavaScript，仅通过 **HttpOnly Cookie** 传递：

```typescript
// auth.controller.ts
@Post('/login')
async login(@Body() body, @Res({ passthrough: true }) res: Response) {
  const result = await this.authService.login(username, password)

  res.cookie('refreshToken', result.refreshToken, {
    httpOnly: true,      // JS 不可访问，防 XSS
    secure: true,        // 仅 HTTPS
    sameSite: 'strict',  // 防 CSRF
    path: '/api/auth',   // 仅鉴权接口携带，减少泄露面
    maxAge: 7 * 24 * 60 * 60 * 1000,
  })

  // 只返回 accessToken，不返回 refreshToken 到 body
  return { accessToken: result.accessToken, user: result.user }
}
```

刷新接口从 Cookie 读取：

```typescript
@Post('/refresh')
async refresh(@Req() req: Request) {
  const token = req.cookies?.refreshToken
  if (!token) throw new UnauthorizedException()
  return this.authService.refresh(token)
}
```

### Refresh Token Rotation（防重放）

每次刷新时，旧 refreshToken **立即作废**，签发全新 token。需数据库记录 token 家族：

```typescript
async refresh(oldToken: string) {
  // 1. 查找 token 家族
  const family = await db.findTokenFamily(oldToken)
  if (!family || family.revoked) {
    // 检测到已作废的 token 被重放 → 该用户所有 token 全部作废（可能已被盗）
    if (family) {
      await db.revokeAllUserFamilies(family.userId)
    }
    throw new UnauthorizedException()
  }

  // 2. 作废当前家族
  await db.revokeFamily(family.id)

  // 3. 签发新 token，创建新家族
  const newTokens = this.generateTokens(user)
  await db.saveTokenFamily(newTokens.refreshToken, user.id)

  return newTokens
}
```

**为什么需要 Rotation？**
- 如果 refreshToken 被盗，攻击者和合法用户都会尝试刷新
- 其中一人先用旧 token 刷新成功 → 旧 token 作废
- 另一人再用同一旧 token 刷新 → 检测到重放 → 整个用户的所有 token 全部作废
- 用户下次需要重新登录，攻击者的 token 也同时失效

## 四、Layer 3 — 会话与设备管理

### Token 黑名单（登出 + 主动吊销）

accessToken 短期有效但无法主动撤销，需配合 Redis 黑名单：

```typescript
// Redis key: "blacklist:accessToken:<jti>" TTL = token 剩余有效时间
async logout(accessToken: string, refreshToken: string) {
  const decoded = jwt.decode(accessToken)
  const ttl = decoded.exp - Math.floor(Date.now() / 1000)

  if (ttl > 0) {
    await redis.set(`blacklist:${decoded.jti}`, '1', 'EX', ttl)
  }
  await db.revokeTokenFamily(refreshToken)

  res.clearCookie('refreshToken', { path: '/api/auth' })
}
```

JWT 验证时需额外检查黑名单：

```typescript
// jwt.strategy.ts → validate()
async validate(payload: { sub: number; username: string; jti: string }) {
  const blacklisted = await redis.get(`blacklist:${payload.jti}`)
  if (blacklisted) throw new UnauthorizedException('Token 已吊销')
  return { userId: payload.sub, username: payload.username }
}
```

### 设备指纹 + 登录历史

```typescript
interface LoginSession {
  id: string
  userId: number
  ip: string
  userAgent: string
  deviceId: string          // 前端生成持久化指纹
  loginAt: Date
  refreshedAt: Date
  expiresAt: Date
}
```

- 新设备首次登录 → 站内信/邮件通知用户
- 异常 IP（异地、VPN）→ 触发二次验证或直接拦截
- 用户可在「设备管理」页面查看并主动撤销任意设备

### 远程踢下线

```typescript
// 管理员或用户主动撤销某设备/全部设备
async revokeSessions(userId: number, sessionId?: string) {
  if (sessionId) {
    await db.revokeSession(sessionId)
  } else {
    await db.revokeAllSessions(userId)
  }
  // 该设备 refreshToken 立即失效
  // 已签发的 accessToken 在下次请求时被黑名单拦截
}
```

## 五、CSRF 防护

| 措施 | 说明 |
|------|------|
| Cookie `sameSite: 'strict'` | 跨站请求不携带 Cookie，最安全 |
| 关键操作加 CSRF Token | 登录/改密/支付等操作，服务端下发一次性的 CSRF Token |
| 自定义 Header 校验 | 前端在 `X-Requested-With: XMLHttpRequest` 头标记，后端校验 |

```typescript
// 如果 Cookie 需要 sameSite: 'lax'（允许顶部导航携带 Cookie）
// 则必须为写操作增加 CSRF Token 双重校验：
@Post('/change-password')
async changePassword(@Body() body, @Headers('x-csrf-token') csrfToken: string) {
  await csrfService.validate(csrfToken, req.cookies)
  // ...
}
```

## 六、SSO 集成（终极方案）

当用户规模增长到需要统一账号体系时，接入 SSO：

```
浏览器 (SPA)
  │
  ├─ 未登录 → SSO 重定向（OIDC / OAuth 2.0）
  │               │
  │               └─ 用户在 SSO 页面认证
  │                  → 回调 → 后端用 code 换 token
  │                  → 签发自有 accessToken（15min）
  │                  → 设置 refreshToken HttpOnly Cookie（7d）
  │
  └─ 后续请求 → Bearer accessToken（同 Layer 1 流程）
```

**优势：**
- 前端零 Token 暴露，完全依赖 HttpOnly Cookie + Bearer
- 账号管理（密码重置、MFA、异地检测）由 SSO 统一提供
- 企业内部可无缝对接  统一认证
