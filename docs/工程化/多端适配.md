# 多端适配方案

> 基于  -chat 项目架构分析，总结一套适合 Demo 项目（React + MobX + Tailwind）的多端适配策略。

---

## 一、核心思路

**CSS 媒体查询改的是"同一个东西长什么样"，JS 响应式改的是"这里应该放什么东西"。**

 -chat 的场景属于后者 — 移动端需要遮罩层、下拉手势、面板互斥，这些不是换个 `width` 或 `position` 能解决的，必须换掉整棵 DOM 子树。

### 对比

| | CSS 媒体查询 | JS 响应式 |
|---|---|---|
| **做什么** | 改**同一 DOM** 的样式 | 换**不同 DOM** 树 |
| **能感知什么** | 视口宽高 | 任何状态（面板开关、拖拽中、全屏中…） |
| **能触发什么** | 无副作用 | 关面板、清状态、发请求… |
| **坐标系** | 纯视觉 | 视觉 + 行为 + 状态 |

## 二、架构概览

```
window.resize 事件
  → GlobalStore.isMobile (MobX observable)
    → 各组件 props 接收 isMobile
      → 分支渲染 (PC DOM 树 vs 移动端 DOM 树)
```

不使用 Tailwind `md:`/`lg:` 响应式断点做布局级切换，而是通过 JS 状态驱动整棵 DOM 树的替换。

### 为什么不用 matchMedia？

`matchMedia` 和 `resize` 回调功能等价，但 `resize` + 手动判断提供更精确的控制：

1. **切换时需要编排副作用顺序** — `isMobile` 变化不只是换样式，还要关闭面板、重置状态、清理 DOM 引用
2. **单一真相源** — `GlobalStore.isMobile` 是唯一判断源，避免 `matchMedia` 回调和 store 更新之间的时序不一致
3. **可手动 throttle** — 控制节流频率

```typescript
// GlobalStore
private handleResize = () => {
  const isMobile = this.checkIsMobile()
  // 只有真正跨越断点时才触发
  if (isMobile !== this.isMobile) {
    this.isMobile = isMobile
    if (isMobile && this.openHistorySidebar) {
      this.closeSidebar()  // 副作用1：关闭侧栏
    }
    // 其他需要重置的状态...
  }
}
```

### 移动端断点

```typescript
const MOBILE_BREAKPOINT_PX = 768
this.isMobile = window.innerWidth < 768
```

## 三、面板的三种形态

一个组件通过 `isMobile` + `position` + `mobilePosition` 覆盖所有端的呈现方式：

| 形态 | 触发条件 | 实现 |
|------|---------|------|
| **PC 侧面板** | `!isMobile` | `fixed` 定位 + 滑入动画 + 可选 resize 手柄 + 可选全屏 |
| **移动端侧滑** | `isMobile && mobilePosition !== 'bottom'` | 全屏黑色遮罩 + `fixed` 面板从左侧/右侧滑入 |
| **移动端底部抽屉** | `isMobile && mobilePosition === 'bottom'` | 从底部弹出 + 下拉拖拽关闭（DOM 直操） |

### PC 与移动端用不同 DOM 树的原因

```
PC 端：没有遮罩层、没有底部弹出、不需要下拉手势
移动端：有遮罩层作为关闭入口、有下拉手势、面板占满宽度
```

这两种情况在 DOM 结构上完全不同，CSS 媒体查询只能改变同一棵 DOM 树的样式，无法增删节点。

## 四、关键设计决策

### 4.1 面板宽度用 `useState`，不放入 MobX Store

**原因：拖拽每秒触发 60 次 `mousemove`，等于每秒 60 次 MobX 广播。**

```
拖拽 1px
  → globalStore.panelWidth = 361
    → MobX 广播到所有订阅组件
      → Layout 重渲染、主内容区重渲染、面板内部组件重渲染
      → ×60次/秒 = 性能灾难
```

放在组件内 `useState`：

```
拖拽 1px
  → setPanelWidth(361)
    → 仅当前组件自身重渲染
    → 主内容区用 flex-1 自适应，不需要知道具体 px 值
```

**原则：高频变更局部化。**

### 4.2 移动端拖拽用 DOM 直操，不走 React 渲染

**React 渲染管线在 60fps 下的瓶颈：**

```
touchmove (每帧 ~16ms)
  → setState → React 调度 → render → commit → 浏览器绘制
    → 中间可能掉帧，出现"卡顿感"
```

**DOM 直操做法：**

```typescript
// 开始拖拽 → 关闭 transition
el.style.transition = 'none'

// 拖拽中 → 直接设置 transform（只触发 GPU 合成层，不走 layout/paint）
el.style.transform = `translateY(${delta}px)`

// 松手 → 判断关闭或弹回
if (delta > 120 || velocity > 0.5) {
  el.style.transform = 'translateY(100%)'  // 关闭动画
  setTimeout(() => onClose?.(), 300)        // 动画播完才通知 React
} else {
  el.style.transform = 'translateY(0px)'    // 弹回动画
}
```

整个拖拽过程 React 完全不参与，只在确定结果后才触发一次状态更新。`transform` 只触发 GPU 合成，保证 60fps 丝滑。

### 4.3 面板互斥 — 同一时刻只有一个面板

```typescript
// 打开右面板 → 强制关左面板
onOpenRight() {
  globalStore.closeSidebar()
}
```

原因：
1. **避免遮罩嵌套** — 移动端面板都有半透明遮罩，两层遮罩 = 极暗 + 层级混乱
2. **避免 z-index 战争** — 两个 `fixed` 面板争层级
3. **用户心智模型简单** — 打开新面板 = 替换旧面板，不是叠加
4. **减少并行状态** — 只有一个面板 open 时，其他面板的订阅不需要响应

## 五、PC 面板的两种定位模式

| 模式 | 定位 | 参照物 | 适用场景 |
|------|------|--------|---------|
| **fixed**（默认） | 相对视口 | 全屏 | 左侧历史面板 — 全局层级，脱离页面文档流 |
| **absolute**（innerMode） | 相对父容器 | 页面内 | 右侧工具面板 — 不挡住顶栏，保持页面上下文完整 |

## 六、Design Token 与设计稿适配

### 为什么设计稿是 2880px？

```
2880px = 1440px × 2 (Retina 2x)
```

Figma 设计稿以 1440px 逻辑分辨率设计，导出 ×2 适配高分屏。

| 设备 | 物理分辨率 | 逻辑分辨率 |
|------|-----------|-----------|
| MacBook Pro 14" | 3024×1964 | 1512×982 |
| MacBook Pro 16" | 3456×2234 | 1728×1117 |
| 常见外接显示器 | 2560×1440 | 1280×720 (HiDPI) |

开发时所有数值 ÷2 再转 rem。

### 三层 Design Token 映射

```
Figma 色值 #3370FF
  → Gradient 色板 CSS 变量：-- -blue-05: #3370ff
    → 语义 Token（Light）：--light-brand-primary: var(-- -blue-05)
    → 语义 Token（Dark）： --dark-brand-primary: var(-- -blue-03)
      → Tailwind class：bg- -brand-primary
```

**好处：**

```html
<!-- ❌ 直接写色值：需要 JS 判断主题 -->
<button className={darkMode ? "bg-blue-300" : "bg-blue-500"}>

<!-- ✅ 语义 Token：CSS 变量随 html.dark 自动切换 -->
<button className="bg- -brand-primary">
```

组件代码不需要感知当前主题，CSS 变量自动切换。对 SSR 和组件库尤为重要。

## 八、设计哲学总结

```
核心矛盾：一套代码 vs 多种设备形态

解决策略：
  不是"一套布局适配所有"   ← CSS 媒体查询思路
  而是"根据条件渲染不同布局" ← JS 状态分支思路

具体手段：
  1. isMobile 状态 → 分支渲染不同 DOM 树（不是不同 CSS）
  2. 面板宽度 → useState（高频变更局部化）
  3. 拖拽动画 → DOM 直操（避开 React 渲染管线）
  4. 面板互斥 → 简化状态、避免层级冲突
  5. Design Token → CSS 变量自动主题切换，组件不感知
```
